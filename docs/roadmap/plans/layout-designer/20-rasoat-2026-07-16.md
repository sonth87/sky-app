## A. Lỗ hổng NGHIỆP VỤ đã phát hiện + hướng chốt

### A1 — Trạng thái `active` của Event: chỉ để theo dõi/lọc, KHÔNG mang nghĩa tự động

**Vấn đề nêu ra:** `status='active'` nằm trong `EventDocument` — lo ngại khi Supabase 2 chiều
sẽ đồng bộ và 2 máy hội trường đè active của nhau.

**Sonth chốt:** `active` ở đây **chỉ mang tính theo dõi/lọc/tìm kiếm**, KHÔNG có nghĩa tự động
(không phải "đến ngày X thì tự chuyển active", không phải "vì đang active nên cái khác không
được active"). **Kể cả không có trạng thái active cũng không sao.** → Không cần tách thành con
trỏ cục bộ phức tạp; giữ `active` như 1 nhãn trạng thái bình thường trong `status`. Vấn đề "2
máy đè nhau" KHÔNG xảy ra vì active không phải cơ chế điều phối, chỉ là nhãn.

**Điểm quan trọng phát sinh từ giải thích của Sonth — cơ chế đồng bộ cloud có 2 kiểu:**
- Dữ liệu (data/config/layout) đã kéo về từ cloud → có **id đồng bộ** app↔cloud → khi cloud có
  bản mới, app chỉ **hiện notice "có bản cập nhật"**, KHÔNG tự động ghi đè.
- Layout có **version** → cần tính năng **publish/draft**: publish thì tự tăng version; tiếp tục
  sửa+lưu mà không publish thì vẫn là nháp → cần **vùng lưu nháp riêng**. Dưới Electron user
  **tự pull về** (không tự động), và có thể **switch version**.
→ Đây là yêu cầu MỚI lớn — tách thành file [21](21-layout-versioning.md).

### A2 — Re-import DataSource: cần MODAL import rõ ràng + màn kết quả (không chỉ hỏi "có import?")

**Vấn đề nêu ra:** re-import cập nhật DataSource consumable đã có `consumedIds` → nếu record đổi
id thì người đã nhận bằng xuất hiện lại ở đợt sau.

**Sonth chốt:** 2 đợt lễ có 2 tình huống: (1) chạy 2 lần khác nhau, data riêng; (2) chạy trên
chính đợt trước + import thêm data vào. → data có thể riêng, cũng có thể chung trên 1 bản data.
Khi import cần **check để ghi đè bản ghi đã có / thêm mới / chỉ nhận data mới** — và phải
**confirm với user qua 1 modal import rõ ràng** (không phải chỉ chạy check rồi hỏi "có import
không" như hiện tại). Modal cần:
- **Màn kết quả import** trên chính modal đó: đã import bao nhiêu, ghi đè bao nhiêu, xóa bao nhiêu.
- **Hiển thị danh sách dạng bảng, highlight màu kiểu git** (thêm/sửa/xóa).
- **id record**: lấy theo 1 trường user quy định (`masv`/`manv`/`id`/...) HOẶC tự sinh id ổn
  định theo khóa tự nhiên.
→ Yêu cầu MỚI — tách thành file [22](22-import-modal.md). Giải luôn lỗ hổng "người đã nhận
xuất hiện lại" vì id ổn định theo khóa tự nhiên → `consumedIds` cũ vẫn trỏ đúng.

### A3 — `consumedIds` ghi khi CHẠY ĐẾN từng người (3 cách), không phải tick tay 500 lần

**Vấn đề nêu ra:** consumedIds thủ công 100% → trao 500 người phải tick tay từng người thì UX
quá cực, dễ bị bỏ luôn không dùng.

**Sonth chốt:** có **3 dạng vận hành** — ấn tay, chạy tự động, quét QR — **chạy đến ai thì đánh
dấu người đó là đã chạy**. Nghĩa là `consumedIds` được ghi **theo luồng vận hành đã có sẵn**
(mỗi khi 1 record được trình chiếu xong / chuyển sang người tiếp theo), không phải thao tác tick
tay tách biệt. Đây KHÔNG mâu thuẫn với "không tự động hóa" đã chốt trước — vì đây là "đánh dấu
theo hành động vận hành thật của người dùng" (họ chủ động cho chạy đến người đó), không phải
"hệ thống tự suy diễn". Cập nhật ở [13](13-ceremony-mo-rong.md) — thay quyết định #15 cũ ("tự
mở tự tick tay hoàn toàn tách khỏi runtime") bằng "đánh dấu theo luồng trình chiếu".

### A4 — Layout đổi sau khi Event map biến → lệch âm thầm (giải bằng versioning A1)

**Vấn đề nêu ra:** designer sửa layout (đổi/thêm token) sau khi Event đã `fieldMap` → runtime
fail-soft nên token mới hiện rỗng lặng lẽ giữa lễ.

**Sonth chốt (qua cơ chế version):** Event **ghim version layout đã chọn**. Khi layout publish
version mới, Event thấy **notice "có bản mới"**. User chọn **Update** → hệ thống **check các
variable đã khớp/map hết chưa**; nếu chưa → thông báo + **hỏi có muốn sang màn map biến không**
(không thì chỉ update). User chọn **không update** → bỏ qua, Event giữ version cũ. → Lỗ hổng
"lệch âm thầm" biến mất vì (1) không tự đổi version giữa lễ, (2) update có bước check token.
Chi tiết ở [21](21-layout-versioning.md) §"Event ↔ layout version".

### A5 — `setActive` giữa lễ đang chạy → phải reset session + backdrop về idle

**Vấn đề nêu ra:** `SessionState.current_on_stage_id` trỏ record của Event A; kích hoạt Event B
thì id thuộc DataSource khác → backdrop kẹt người của đợt cũ.

**Chốt (đề xuất, chưa cần Sonth quyết — logic thuần):** `setActive(id)` mà id khác Event đang
chạy → reset `SessionState` (current/pending về null) + đưa backdrop về màn chờ (idle). Ghi vào
phạm vi GĐ3. Đây là hệ quả kỹ thuật tất yếu, không phải lựa chọn nghiệp vụ.

## B. Vấn đề KỸ THUẬT đã phát hiện + hướng chốt

### B1 — Web WASM: nhiều tab đè dữ liệu (control + backdrop là 2 tab!)

Mỗi tab có 1 bản DB trong memory, save = ghi đè toàn bộ IndexedDB → tab sau đè tab trước không
cảnh báo. → Tối thiểu: **Web Locks API** giữ quyền ghi 1 tab, tab còn lại read-only + cảnh báo.
Ghi vào [18](18-luu-tru-sqlite-supabase.md) §1a như giới hạn đã biết + việc phải làm khi web
mode dùng thật. Không chặn GĐ0 (GĐ0 chưa có editor web đa-tab thật).

### B2 — IndexedDB bị trình duyệt evict → mất dữ liệu

WASM mode lưu IndexedDB, trình duyệt được quyền xóa khi thiếu dung lượng. → Gọi
`navigator.storage.persist()` + nhắc export định kỳ. Việc nhỏ, ghi vào file 18 + phạm vi bước
WASM.

### B3 — Ảnh không preload → backdrop nháy trắng giữa lễ

`LayoutRenderer` mới render ảnh nền/avatar theo record; thiếu preload ảnh record kế tiếp thì mỗi
lần gọi tên có khoảnh khắc trắng. → Đưa "preload ảnh record kế tiếp + ảnh nền các variant của
layout active" vào DoD GĐ1/GĐ4d.

### B4 — Editor cần kiến trúc registry đầy đủ (undo/redo/history/zoom/helper-line/snap/drag-drop)

**Sonth yêu cầu rõ:** editor phải có **undo, redo, history, zoom, helper line, snap, drag drop,
canvas... cần 1 kiến trúc hợp lý để đăng ký, quản lý registry đầy đủ cho layout.** → KHÔNG cắt
khỏi v1. Tách **package `editor-core` riêng** — file [23](23-editor-core-architecture.md).

### B5 — Thử `node:sqlite` thay `better-sqlite3` (bỏ ABI dance)

Node ≥22.5 có `node:sqlite` builtin (không phải native addon → không rebuild ABI 2 chiều). Kiến
trúc `SqlExecutor` cho phép đổi driver không đụng query. → **Thử nghiệm đầu GĐ1** (Sonth chốt):
kiểm chứng Electron 43 bundle Node ≥22.5 + API đủ dùng. Ổn thì bỏ `better-sqlite3`. Không ổn thì
giữ (không mất gì). Ghi vào plan GĐ1.

### B6 — Trùng lặp code seed/mapping 3 nơi (do chính GĐ0)

`mapStatus`/`mapRawStudent`/`defaultCeremony`/`defaultConfig` giờ tồn tại ở 3 nơi (Electron
sync.ts, data-service store.ts, sqlite-wasm-data.ts). → Gom về `ceremony-db` (hoặc slide-shared)
trong đợt dọn GĐ0. Xem phần "Sửa GĐ0" trong plan.

### B7 — Các nợ nhỏ (ghi nhận, xử lý dần)
- Validate schema khi import bundle (file JSON user sửa tay được) — zod/guard, phạm vi GĐ5.
- `replaceStudents` prepare lại statement mỗi dòng — nợ hiệu năng có chủ đích (10k+ record mới thấy).
- `DataStore` interface hiện là trang trí (không adapter nào implement) — GĐ1-3 phải dùng thật.
- Version-coupling `sql.js` (wasm từ shell-web, JS từ ceremony-db) — pin exact version.
- Executor không `close()` lúc quit — checkpoint WAL lúc `will-quit`.
- `preload` ảnh (B3 ở trên).
